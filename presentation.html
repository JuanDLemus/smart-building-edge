Arquitectura del Sistema

Una topología Edge-to-Cloud diseñada para baja latencia.

  • Sensado en el Edge: ESP32 como nodos de recolección de datos ambientales.
  • Procesamiento Local: Una Raspberry Pi 5 actúa como Gateway y Broker.
  • Inteligencia Distribuida: Inferencias de Machine Learning ejecutadas in-situ.
  • Disponibilidad: Redes privadas empresariales (Tailscale) y exposición controlada (ngrok/Vercel).

Malla de Nodos ESP32

Hardware y Sensores

  • Microcontrolador: ESP32 NodeMCU, elegido por su conectividad Wi-Fi nativa bidireccional.
  • Módulo de Gases (MQ135): Detecta CO2, Amoniaco, Benceno y alcohol, vitales para determinar la calidad del aire de las aulas.
  • Sensores Atmosféricos: Medición en tiempo real de Temperatura, Humedad y Presión barométrica.
  • Comunicación: Publicación asíncrona hacia el servidor MQTT sin bloquear el ciclo de medición físico.

Gestión de Red y Seguridad

Acceso Seguro (Tailscale)

Implementamos una VPN de malla (WireGuard) para que los administradores y profesores puedan acceder al sistema como si estuvieran en la red local de la universidad, garantizando encriptación de extremo a extremo y evitando vulnerabilidades de puertos abiertos.

Acceso Invitado (Ngrok/Vercel)

Para estudiantes o personal externo que requiera ver el estado de ocupación, el frontend del dashboard se expone temporalmente mediante túneles inversos o deploys estáticos web, aislando y protegiendo la base de datos principal de accesos no autorizados.

Despliegue con Docker (QEMU)

¿Por qué virtualizar en la Raspberry Pi 5?

  • Entorno Aislado: Evita conflictos irreversibles de dependencias entre las librerías científicas de Python, Scikit-Learn y los paquetes nativos del SO base (Debian/Raspbian).
  • Portabilidad Absoluta: El código corre exactamente igual en las computadoras Windows de los desarrolladores y en el hardware Linux final del laboratorio.
  • Emulación Cruzada (QEMU): Permite construir imágenes ARM64 en computadoras de arquitectura x86_64, facilitando enormemente la distribución.

Orquestación: docker-compose.yml

Múltiples contenedores interactúan bajo una red virtual de puente (bridge local).

  • mqtt: Usa la imagen oficial eclipse-mosquitto:latest. Responsable de la ingesta en el puerto 1883.
  • web: El core de procesamiento en Flask.
  • Persistencia de Datos: Se utilizan volúmenes mapeados (edge-data) para garantizar que la base de datos SQLite retenga su histórico frente a caídas eléctricas o reinicios de contenedor.

services:
  mqtt:
    image: eclipse-mosquitto:latest
    ports:
      - "1883:1883"
  web:
    build: .
    ports:
      - "5001:5000"
    environment:
      - MQTT_BROKER=mqtt
      - DB_PATH=/app/data/building.db

Empaquetado: Dockerfile

Receta inmutable para la construcción de la imagen base.

  • SO Base Ligero: python:3.11-slim reduce drásticamente el peso de la imagen y la superficie de ataque, optimizado para recursos limitados.
  • Inyección del Modelo: El modelo serializado occupancy_model.pkl se pre-empaqueta en la imagen.
  • Inicialización SQL: La ejecución temprana del script init_db.py garantiza la creación limpia de tablas SQL antes de abrir sockets.

FROM python:3.11-slim
WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

RUN mkdir -p /app/data
RUN python init_db.py /app/data/building.db

EXPOSE 5000
CMD ["python", "app.py"]

Implementación de Machine Learning

Predicción de Ocupación mediante Análisis Multivariable

A diferencia de sistemas mecánicos simples, este proyecto no depende ciegamente de un láser contador de puertas. Toma lecturas correlacionadas de CO2, Temperatura y Humedad relativa simultáneamente. Estas variables cambian de manera predecible según el número de ocupantes vivos que respiran e irradian calor corporal en espacios cerrados (aulas / laboratorios).

Evaluación de Algoritmos (Métricas)

Se evaluó exhaustivamente un abanico de estimadores estadísticos y árboles de decisión (Regresión y Clasificación). Las curvas de aprendizaje y validación cruzada nos permitieron seleccionar el modelo capaz de generalizar los datos sin incurrir en memorización (Overfitting).

Elección Definitiva: Random Forest

¿Por qué este algoritmo?

  • Inmunidad al Ruido (Resiliencia): Los sensores analógicos económicos sufren de ruido térmico y picos de voltaje. Un ensamble de múltiples árboles dispersos absorbe excelentemente los outliers.
  • Costo Computacional vs XGBoost: Aunque XGBoost es poderoso, Random Forest es sumamente eficiente y veloz haciendo inferencias asíncronas sobre procesadores de la familia ARM.
  • Pre-procesamiento Minimalista: No demanda esquemas estrictos de normalización ni escalamiento (como MinMax), a diferencia de implementaciones complejas con Redes Neuronales Profundas.

Estándares de Comunicación

MQTT (Capa de Máquinas)

En lugar de agotar el Wi-Fi realizando HTTP Polling (preguntar cíclicamente), aplicamos el patrón Publish/Subscribe. Los microcontroladores publican strings JSON ultra-ligeros. El broker Mosquitto enruta los mensajes instantáneamente al backend, logrando una reducción superior al 80% en el consumo de ancho de banda y batería de los nodos.

WebSockets (Capa de Usuarios)

Para reflejar alertas en el Dashboard en tiempo real (menos de 50ms de latencia) sin que el cliente recargue la página, implementamos la librería Socket.IO. Flask empuja un evento (Push) new_data hacia todos los navegadores conectados una fracción de segundo posterior al cálculo de inferencia algorítmica.

Limitaciones Encontradas

Desafíos: Hardware y Redes

  • Calibración Extrema del sensor MQ135:
    Experimentamos gran dificultad en la calibración física de su resistor variable (RL). Los índices de resistencia cambiaban drásticamente dependiendo de la humedad y presión climática, arrojando alarmas de polución que eran falsos positivos en climas muy fríos.
  • Inestabilidad Wi-Fi Malla Institucional:
    La infraestructura empresarial expulsaba frecuentemente a los ESP32, por lo que fue imperativo programar rutinas de reconexión profunda para evitar bloqueos por tiempo de espera en el Watchdog Timer.
Limitaciones Encontradas

Desafíos: Data Pipeline

  • Etiquetado Humano Exhaustivo:
    Recolectar el set de datos para entrenar el modelo implicó contar la ocupación visualmente por horas. Esta carga manual inyectó sesgos inevitables y errores temporales de desincronización con el hardware.
  • Inercia de Partículas y Gases:
    Existe un problema físico termodinámico. El CO2, el calor y la humedad no desaparecen milisegundos después de que las personas evacúan el aula. Existe una latencia prolongada (decaimiento exponencial) que confundió al modelo momentáneamente.
