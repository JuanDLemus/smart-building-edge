Arquitectura del Sistema

Una topología Edge-to-Cloud distribuida con concurrencia asíncrona.

  • Edge Computing: Microcontroladores operando como nodos de lectura y publicación (Publishers).
  • Gateway & Broker: Raspberry Pi 5 actuando como hub de enrutamiento MQTT local (Mosquitto).
  • Backend Híbrido: Servidor Flask ejecutando hilos paralelos (Eventlet) para ML en tiempo real.
  • Zero-Trust Network: Acceso segmentado mediante Tailscale (UDP Hole Punching) para eludir NAT institucionales.

Complejidad en el Edge (ESP32)

Implementación en C++ enfocada en alta disponibilidad y bajo bloqueo (Non-blocking I/O).

  • Hardware Exacto: Sensores analógicos MQ135 y MQ8 procesados mediante ADCs de 12-bits.
  • Filtro Matemático: Se calculan deltas base tras 100 muestras de calibración inicial.
  • JSON Parsing: Uso de ArduinoJson para empacar los datos estructurados dinámicamente antes de enviar.
// Loop No-Bloqueante (sin delay) con JSON
unsigned long tiempoActual = millis();
if (tiempoActual - tiempoAnterior >= 2000) {
  float mq135_delta = analogRead(MQ135_PIN) - base_135;
  float mq8_delta = analogRead(MQ8_PIN) - base_8;
  float co2_index = (mq135_delta*0.9) + (mq8_delta*0.1);

  StaticJsonDocument<200> doc;
  doc["mq135"] = mq135_raw;
  doc["co2_index"] = max(0.0, co2_index);
  
  String payload;
  serializeJson(doc, payload);
  mqttClient.publish("salon/LAB1/gases", payload.c_str());
}

Arquitectura de Acceso Seguro

Red Empresarial (Tailscale VPN)

La Raspberry Pi 5 opera dentro de la intranet de la Universidad. Para lograr acceso global sin abrir puertos físicos en el router universitario, desplegamos Tailscale (basado en WireGuard). Esto crea un túnel NAT encriptado Peer-to-Peer exclusivo para desarrolladores y el Profesor.

Proxy Público Inverso

Dado que el cliente final (estudiantes) no debe instalar VPNs, el puerto 5001 del Dashboard se expone mediante túneles inversos (Ngrok) o despliegues serverless (Vercel). Esto mitiga el riesgo de ataques DDoS directos hacia el hardware del proyecto.

Backend de Alta Concurrencia (Python)

Superamos la limitación del GIL de Python (Global Interpreter Lock) usando subprocesos asíncronos.

  • Eventlet Monkey-Patching: Permite que Flask gestione miles de conexiones WebSocket concurrentes.
  • Daemon Threads: El loop del cliente MQTT (mqtt_client.loop) se ejecuta en un background_task aislado para no congelar el servidor web.
  • Wildcards MQTT: Suscripción dinámica a salon/+/gases en un solo socket.

import eventlet
eventlet.monkey_patch() # Habilita concurrencia real

def handle_mqtt_message(msg):
    payload = json.loads(msg.payload.decode())
    # ... inferencia ML en vivo ...
    socketio.emit("new_data", { "lab": lab_name, ... })

def run_mqtt():
    mqtt_client.subscribe("salon/+/gases") # Wildcard topic
    while True:
        mqtt_client.loop(timeout=1.0)
        socketio.sleep(0.1) # Cede el hilo al framework web

if __name__ == "__main__":
    socketio.start_background_task(target=run_mqtt)
    socketio.run(app, host="0.0.0.0", port=5000)

Virtualización ARM en Edge (QEMU)

Abstracción sobre Raspberry Pi 5

  • Entorno Inmutable: Evita el infierno de dependencias de C++ para compilar Scikit-Learn directamente en Debian/Raspbian ARM64.
  • Emulación Cruzada (BuildX): Permite empaquetar binarios nativos ARM desde los portátiles x86 de los desarrolladores.
  • Orquestación Compose: Un archivo unifica el ciclo de vida del broker C puro (Eclipse Mosquitto) junto con el contenedor Python de Flask.

Inteligencia Distribuida (Pipeline)

Inferencia en Streaming

El sistema procesa inferencias continuas en el callback MQTT, no en lotes (Batch). En milisegundos, el vector de variables [MQ135, MQ8, CO2_Index] se inyecta en la función predict_occupancy() de Scikit-Learn que retorna el estado exacto (VACÍO, PARCIAL, OCUPADO) antes de insertarse en SQLite.

Evaluación de Algoritmos (Métricas)

Se evaluó exhaustivamente un abanico de estimadores estadísticos y árboles de decisión (Regresión y Clasificación). Las curvas de aprendizaje y validación cruzada nos permitieron seleccionar el modelo capaz de generalizar los datos sin incurrir en memorización (Overfitting).

Elección Definitiva: Random Forest

¿Por qué este algoritmo?

  • Inmunidad al Ruido (Resiliencia): Los sensores analógicos económicos sufren de ruido térmico y picos de voltaje. Un ensamble de múltiples árboles dispersos absorbe excelentemente los outliers.
  • Costo Computacional vs XGBoost: Aunque XGBoost es poderoso, Random Forest es sumamente eficiente y veloz haciendo inferencias asíncronas sobre procesadores de la familia ARM.
  • Pre-procesamiento Estático: No requiere costosas capas de escalamiento tensorial, ahorrando valiosos ciclos de CPU en la Raspberry Pi 5.

Estándares de Comunicación

MQTT (Capa de Máquinas)

En lugar de agotar el Wi-Fi realizando HTTP Polling (preguntar cíclicamente), aplicamos el patrón Publish/Subscribe. Los microcontroladores publican strings JSON ultra-ligeros. El broker Mosquitto enruta los mensajes instantáneamente al backend, logrando una reducción superior al 80% en el consumo de ancho de banda y batería de los nodos.

WebSockets (Capa de Usuarios)

Para reflejar alertas en el Dashboard en tiempo real (menos de 50ms de latencia) sin que el cliente recargue la página, implementamos la librería Socket.IO. Flask empuja un evento (Push) new_data hacia todos los navegadores conectados una fracción de segundo posterior al cálculo de inferencia algorítmica.

Limitaciones Encontradas

Desafíos: Hardware y Redes

  • Calibración Extrema del sensor MQ135:
    Experimentamos gran dificultad en la calibración física de su resistor variable (RL). Los índices de resistencia cambiaban drásticamente dependiendo de la humedad y presión climática. Sensores alternativos a explorar: El BME680 de Bosch o el SGP30 de Sensirion, los cuales integran medición de eCO2 (dióxido de carbono equivalente) compensado térmicamente de fábrica, entregando lecturas digitales estables sobre bus I2C en lugar de caídas de voltaje analógicas propensas a ruido.
  • Inestabilidad Wi-Fi Malla Institucional:
    La infraestructura empresarial expulsaba frecuentemente a los ESP32, por lo que fue imperativo programar rutinas de reconexión profunda.
  • Tiempos Muertos sin Detección de Movimiento:
    Para optimizar aún más la predicción de "AULA VACÍA", se podrían fusionar los datos ambientales con sensores alternativos de presencia de bajo costo: como el radar de ondas milimétricas LD2410 (mmWave), capaz de detectar la micro-respiración humana estática, a diferencia de los sensores PIR tradicionales que fallan si los estudiantes están sentados sin moverse.
Limitaciones Encontradas

Desafíos: Data Pipeline

  • Etiquetado Humano Exhaustivo:
    Recolectar el set de datos para entrenar el modelo implicó contar la ocupación visualmente por horas. Esta carga manual inyectó sesgos inevitables y errores temporales de desincronización con el hardware.
  • Inercia de Partículas y Gases:
    Existe un problema físico termodinámico. El CO2, el calor y la humedad no desaparecen milisegundos después de que las personas evacúan el aula. Existe una latencia prolongada (decaimiento exponencial) que confundió al modelo momentáneamente.

Posibilidades de Mejora: Arquitectura Lógica

Contrarrestar Concept Drift

Introducir un MLOps en línea (Online Machine Learning) capaz de asimilar fluctuaciones climáticas de largo plazo ajustando los pesos del árbol automáticamente para no perder precisión.

TinyML: Inferencia en el Microcontrolador

Exportar el modelo Scikit-Learn entrenado a TensorFlow Lite for Microcontrollers (TFLite). Esto permitiría a los propios ESP32 ejecutar la inferencia localmente sin Wi-Fi, reportando únicamente el cambio de estado (Vacío/Ocupado) al servidor, reduciendo el tráfico MQTT en un 90%.

Posibilidades de Mejora: Hardware e Infraestructura

Transición a Energía Solar (Deep Sleep)

Habilitar modos profundos (Deep Sleep) en los núcleos del ESP32 para consumir microamperios. Acoplar reguladores MPPT (Maximum Power Point Tracking) con paneles solares para recargar baterías 18650, posibilitando despliegues de pared sin enchufes.

MQTT Cloud Bridging

Configurar un enlace de puente (Bridge) en Mosquitto local para retransmitir asíncronamente los históricos agregados hacia brokers en la nube corporativa (ej. AWS IoT Core) posibilitando análisis de Big Data y retención multi-anual sin saturar la tarjeta SD de la Raspberry Pi.
